**Business Requirements Document (BRD)**

**Project Name:** Credit Card Usage Analysis & Promotion Targeting

**Version:** 1.0

**Date:** 2024-07-26

**Author:** AI Someone

**1. Introduction / Overview**

*   **1.1 Purpose:** เอกสารนี้กำหนดความต้องการทางธุรกิจสำหรับโครงการวิเคราะห์พฤติกรรมการใช้บัตรเครดิต เพื่อให้ได้ข้อมูลเชิงลึกเกี่ยวกับรูปแบบการใช้จ่ายของลูกค้ากลุ่มต่างๆ ระบุหมวดหมู่ร้านค้าและค่าธรรมเนียมที่สำคัญ และเพื่อระบุกลุ่มลูกค้าเป้าหมายที่เหมาะสมสำหรับแคมเปญโปรโมชั่นพิเศษ
*   **1.2 Project Goal:** พัฒนาระบบหรือโมดูลที่สามารถประมวลผลข้อมูลธุรกรรมบัตรเครดิตและข้อมูลลูกค้า เพื่อสร้างรายงานการวิเคราะห์พฤติกรรม และระบุโอกาสในการส่งเสริมการขายที่ตรงเป้าหมายมากขึ้น
*   **1.3 Target Audience for this BRD:** Business Stakeholders, Marketing Department, Product Management, Data Analytics Team, Technical Leads.

**2. Business Objectives & Goals**

*   **BO1:** ทำความเข้าใจรูปแบบการใช้จ่าย (Spending Patterns) ของลูกค้าแต่ละกลุ่ม (Segment) เพื่อการวางแผนกลยุทธ์ทางธุรกิจและการตลาดที่มีประสิทธิภาพ
*   **BO2:** ระบุประเภทหมวดหมู่ร้านค้า (Merchant Category Code - MCC) ที่ลูกค้าใช้จ่ายบ่อยที่สุด เพื่อเพิ่มประสิทธิภาพในการสร้างพันธมิตรทางธุรกิจ หรืออกโปรโมชั่นเฉพาะหมวดหมู่
*   **BO3:** ระบุหมวดหมู่หรือประเภทธุรกรรมที่สร้างค่าธรรมเนียมสูงสุด เพื่อทำความเข้าใจแหล่งรายได้ หรือพิจารณาโอกาสในการสร้างโปรโมชั่นที่เกี่ยวข้องกับค่าธรรมเนียม (เช่น Fee waiver)
*   **BO4:** เพิ่มประสิทธิภาพของแคมเปญโปรโมชั่น โดยการระบุกลุ่มลูกค้าที่มีแนวโน้มจะสนใจข้อเสนอพิเศษต่างๆ ได้อย่างแม่นยำมากขึ้น เพิ่มอัตราการตอบรับ (Conversion Rate) และความผูกพันของลูกค้า (Engagement)

**3. Scope**

*   **3.1 In-Scope:**
    *   การนำเข้า (Ingestion) และประมวลผลข้อมูลธุรกรรมบัตรเครดิต, ข้อมูลลูกค้า, และข้อมูลหมวดหมู่ร้านค้า (MCC Mapping) จากแหล่งข้อมูลภายในที่กำหนด
    *   การแบ่งกลุ่มลูกค้า (Customer Segmentation) ตามเกณฑ์ที่กำหนด (เช่น ยอดใช้จ่าย, ความถี่, หมวดหมู่ที่ใช้บ่อย)
    *   การวิเคราะห์และคำนวณ:
        *   หมวดหมู่ร้านค้า (MCC) ที่ใช้บ่อยที่สุด (ตามความถี่/จำนวนเงิน) ต่อลูกค้า/กลุ่มลูกค้า
        *   ยอดรวมค่าธรรมเนียมที่เกิดขึ้น แยกตามหมวดหมู่/ประเภทธุรกรรม ต่อลูกค้า/กลุ่มลูกค้า
    *   การพัฒนากลไก (Logic) เพื่อระบุลูกค้าที่อาจเหมาะสมกับโปรโมชั่นพิเศษตามเกณฑ์ที่กำหนด
    *   การสร้างรายงานสรุปผลการวิเคราะห์ตามที่ระบุในหัวข้อ Reporting Requirements
    *   การจัดเก็บผลลัพธ์การวิเคราะห์และรายชื่อลูกค้าเป้าหมายในรูปแบบที่กำหนด

*   **3.2 Out-of-Scope:**
    *   การพัฒนาระบบ User Interface (UI) สำหรับให้ผู้ใช้ทางธุรกิจเข้าถึงหรือ Query ข้อมูลดิบโดยตรง (เน้นที่ผลลัพธ์ในรูปแบบรายงาน)
    *   การวิเคราะห์ข้อมูลแบบ Real-time (โครงการนี้เน้นการประมวลผลแบบ Batch เช่น รายวัน หรือ รายสัปดาห์)
    *   การเชื่อมต่อหรือพัฒนาระบบสำหรับ *ส่ง* โปรโมชั่นไปยังลูกค้าโดยตรง (โครงการนี้เน้นการ *ระบุ* กลุ่มเป้าหมาย)
    *   การนำเข้าข้อมูลจากแหล่งภายนอกที่ไม่ได้ระบุไว้ (เช่น ข้อมูล Credit Score จากภายนอก เว้นแต่จะกำหนดเพิ่มเติม)
    *   การเปลี่ยนแปลง Logic การคิดค่าธรรมเนียมบัตรเครดิต

**4. Stakeholders**

*   Marketing Department: ผู้ใช้งานหลักของรายงานและรายชื่อลูกค้าเป้าหมาย
*   Product Management: ผู้กำหนดทิศทางและฟีเจอร์ของระบบ
*   Data Analytics / Business Intelligence (BI) Team: ผู้พัฒนาและดูแลรักษาระบบ/โมดูลการวิเคราะห์
*   IT / Data Engineering Team: ผู้ดูแลและจัดเตรียมแหล่งข้อมูลและ Infrastructure
*   (อาจมี) Risk Management: หากการวิเคราะห์ค่าธรรมเนียมเกี่ยวข้องกับการบริหารความเสี่ยง

**5. Functional Requirements (FR)**

*   **FR01: Data Ingestion & Preparation**
    *   FR01.1: ระบบต้องสามารถดึงข้อมูลธุรกรรมบัตรเครดิตจาก Transaction Log Database (หรือ Data Warehouse) ที่มีความถี่ตามกำหนด (เช่น รายวัน)
    *   FR01.2: ระบบต้องสามารถดึงข้อมูลลูกค้าล่าสุดจาก Customer Master Database (หรือ CRM)
    *   FR01.3: ระบบต้องสามารถเข้าถึงตาราง Merchant Category Code (MCC) Mapping เพื่อแปลง MCC เป็นชื่อหมวดหมู่ที่เข้าใจง่าย
    *   FR01.4: ระบบต้องสามารถทำความสะอาด (Cleanse) และแปลง (Transform) ข้อมูลให้อยู่ในรูปแบบที่พร้อมสำหรับการวิเคราะห์ (เช่น จัดการ Missing Values, จัดรูปแบบวันที่)

*   **FR02: Customer Segmentation**
    *   FR02.1: ระบบต้องสามารถแบ่งกลุ่มลูกค้าตามเกณฑ์ที่กำหนดได้ อย่างน้อย 3-5 กลุ่ม (เช่น กลุ่ม High Spender, กลุ่ม Frequent Online Shopper, กลุ่ม New Customer, กลุ่ม Low Activity)
    *   FR02.2: เกณฑ์การแบ่งกลุ่มต้องสามารถปรับเปลี่ยนได้ในอนาคต (Configurable segmentation rules)
        *   *ตัวอย่างเกณฑ์:* ยอดใช้จ่ายรวม 3 เดือนล่าสุด, ความถี่ในการใช้บัตร, MCC ที่ใช้บ่อย, อายุการเป็นลูกค้า, ประเภทบัตรที่ถือ

*   **FR03: Spending Pattern Analysis**
    *   FR03.1: สำหรับลูกค้าแต่ละรายและแต่ละกลุ่มลูกค้า, ระบบต้องสามารถระบุ Top N หมวดหมู่ร้านค้า (MCC) ที่มีการใช้จ่าย *บ่อยที่สุด* (นับจำนวน Transaction) ในช่วงเวลาที่กำหนด (เช่น 3 เดือนล่าสุด)
    *   FR03.2: สำหรับลูกค้าแต่ละรายและแต่ละกลุ่มลูกค้า, ระบบต้องสามารถระบุ Top N หมวดหมู่ร้านค้า (MCC) ที่มี *ยอดใช้จ่ายรวมสูงสุด* ในช่วงเวลาที่กำหนด

*   **FR04: Fee Analysis**
    *   FR04.1: สำหรับลูกค้าแต่ละรายและแต่ละกลุ่มลูกค้า, ระบบต้องสามารถคำนวณ *ยอดรวมค่าธรรมเนียม* ที่เกิดขึ้น แยกตามหมวดหมู่ร้านค้า (MCC) หรือประเภทธุรกรรม (เช่น กดเงินสด, รูดต่างประเทศ) ในช่วงเวลาที่กำหนด
    *   FR04.2: ระบบต้องสามารถระบุ Top N หมวดหมู่ร้านค้า (MCC) หรือประเภทธุรกรรม ที่สร้าง *ค่าธรรมเนียมรวมสูงสุด* ให้กับธนาคาร/บริษัท ในช่วงเวลาที่กำหนด

*   **FR05: Promotion Candidate Identification**
    *   FR05.1: ระบบต้องสามารถระบุรายชื่อลูกค้า (เช่น Customer ID) ที่มีคุณสมบัติตรงตามเกณฑ์ที่กำหนดสำหรับโปรโมชั่นพิเศษ
    *   FR05.2: เกณฑ์การระบุต้องสามารถกำหนดและปรับเปลี่ยนได้
        *   *ตัวอย่างเกณฑ์:*
            *   ลูกค้าในกลุ่ม High Spender ที่ใช้จ่ายในหมวดหมู่ X (เช่น ท่องเที่ยว) สูง แต่ยังไม่เคยได้รับโปรโมชั่น Y (เช่น ส่วนลดโรงแรม)
            *   ลูกค้าที่จ่ายค่าธรรมเนียมประเภท Z (เช่น ค่าธรรมเนียมรายปี) สูง อาจเป็นเป้าหมายสำหรับโปรโมชั่นยกเว้นค่าธรรมเนียม
            *   ลูกค้าที่มี Pattern การใช้จ่ายคล้ายคลึงกับกลุ่มลูกค้าที่ตอบสนองดีต่อแคมเปญก่อนหน้า
            *   ลูกค้าที่ใช้จ่ายในหมวดหมู่ A บ่อย แต่ยอดใช้จ่ายในหมวดหมู่ B (ที่ต้องการโปรโมท) ยังน้อย

*   **FR06: Report Generation**
    *   FR06.1: ระบบต้องสามารถสร้าง "รายงานสรุปพฤติกรรมตามกลุ่มลูกค้า" (Segment Behavior Summary Report) ซึ่งประกอบด้วยข้อมูลอย่างน้อย:
        *   จำนวนลูกค้าในแต่ละกลุ่ม
        *   ค่าเฉลี่ยยอดใช้จ่ายต่อลูกค้าในกลุ่ม
        *   Top 5 MCCs ที่ใช้บ่อยที่สุด (ตามความถี่และยอดใช้จ่าย) ของกลุ่ม
        *   Top 3 หมวดหมู่/ประเภทธุรกรรมที่สร้างค่าธรรมเนียมสูงสุดของกลุ่ม
    *   FR06.2: ระบบต้องสามารถสร้าง "รายชื่อลูกค้าเป้าหมายสำหรับโปรโมชั่น" (Promotion Candidate List) ซึ่งประกอบด้วยข้อมูลอย่างน้อย:
        *   Customer ID
        *   Segment ของลูกค้า
        *   เหตุผล/เกณฑ์ที่ทำให้ลูกค้าถูกเลือก (เช่น "High spender in Travel", "High Annual Fee Paid")
        *   (Optional) คะแนนความเหมาะสม (Propensity Score) ถ้ามีการคำนวณ
    *   FR06.3: รายงานควรสามารถ Export เป็น Format มาตรฐานได้ (เช่น CSV, Excel)

**6. Non-Functional Requirements (NFR)**

*   **NFR01: Performance:** การประมวลผลข้อมูลและสร้างรายงานแบบ Batch (รายวัน) ควรเสร็จสิ้นภายในระยะเวลาที่กำหนด (เช่น ภายใน 4 ชั่วโมงหลังได้รับข้อมูลครบถ้วน)
*   **NFR02: Scalability:** ระบบต้องออกแบบมาเพื่อรองรับปริมาณข้อมูลธุรกรรมและจำนวนลูกค้าที่เพิ่มขึ้นในอนาคต (เช่น รองรับการเติบโต 20% ต่อปี) โดยไม่ทำให้ Performance ลดลงอย่างมีนัยสำคัญ
*   **NFR03: Data Accuracy & Consistency:** ผลการคำนวณและรายงานต้องมีความถูกต้องแม่นยำ ต้องมีกลไกตรวจสอบความถูกต้องของข้อมูลและการคำนวณ
*   **NFR04: Security & Privacy:** ข้อมูลที่ใช้ (ธุรกรรม, ลูกค้า) เป็นข้อมูลส่วนบุคคลที่ละเอียดอ่อน ระบบต้องปฏิบัติตามนโยบายความปลอดภัยของข้อมูลและกฎหมายคุ้มครองข้อมูลส่วนบุคคล (เช่น PDPA) อย่างเคร่งครัด มีการควบคุมการเข้าถึงข้อมูลและผลลัพธ์อย่างเหมาะสม
*   **NFR05: Maintainability:** โค้ดและ Logic การคำนวณต้องมีการจัดการที่ดี มีเอกสารประกอบ (Documentation) ที่ชัดเจน เพื่อให้ง่ายต่อการแก้ไขและปรับปรุงในอนาคต

**7. Data Requirements**

*   **7.1 Transaction Data:** ต้องการฟิลด์อย่างน้อย: Transaction ID, Customer ID, Card ID (ถ้ามี), Merchant ID/Name, Merchant Category Code (MCC), Transaction Timestamp, Transaction Amount, Currency, Fee Amount (ถ้ามีแยก), Transaction Type (Purchase, Cash Advance, etc.)
*   **7.2 Customer Data:** ต้องการฟิลด์อย่างน้อย: Customer ID, Customer Segment (ถ้ามีอยู่แล้ว), Join Date, (Optional: Demographic data like Age Group, Location for segmentation)
*   **7.3 Merchant Category Data:** ต้องการตาราง Mapping: MCC, Category Name (e.g., "Grocery Stores", "Airlines", "Restaurants"), Category Group (e.g., "Travel", "Retail", "Dining")

**8. Reporting Requirements**

*   **RPT01:** Segment Behavior Summary Report (ตาม FR06.1) - ความถี่: รายสัปดาห์/รายเดือน, Format: Excel/PDF, ผู้รับ: Marketing, Product Mgmt
*   **RPT02:** Promotion Candidate List (ตาม FR06.2) - ความถี่: รายสัปดาห์/ตามแคมเปญ, Format: CSV/Secure File Transfer, ผู้รับ: Marketing Campaign Team

**9. Assumptions**

*   แหล่งข้อมูล (Transaction, Customer, MCC) มีอยู่และสามารถเข้าถึงได้
*   ข้อมูลจากแหล่งต่างๆ มีคุณภาพเพียงพอสำหรับการวิเคราะห์ (มีความสมบูรณ์, ถูกต้องในระดับที่ยอมรับได้)
*   มีคำจำกัดความที่ชัดเจนสำหรับ Merchant Category Code (MCC) และการ Mapping ที่ถูกต้อง
*   ทีม Marketing และ Product จะสามารถให้คำจำกัดความและเกณฑ์สำหรับ Customer Segments และ Promotion Candidates ได้
*   มี Infrastructure (เช่น Database, Processing Engine) ที่เพียงพอสำหรับการดำเนินโครงการ

**10. Constraints**

*   ข้อจำกัดด้านงบประมาณและระยะเวลาของโครงการ (ระบุ TBD หรือค่าที่ทราบ)
*   ข้อจำกัดด้านเทคโนโลยี (อาจต้องใช้ Technology Stack ที่มีอยู่ภายในองค์กร)
*   ข้อบังคับด้านกฎหมายและนโยบายภายในเกี่ยวกับการใช้ข้อมูลลูกค้า (PDPA Compliance)

**11. Success Metrics**

*   **SM1:** สามารถสร้างรายงาน RPT01 และ RPT02 ได้ถูกต้อง ตรงตามกำหนดเวลา และครบถ้วนตาม Requirements
*   **SM2:** ทีม Marketing นำข้อมูลเชิงลึกจากรายงานไปใช้วางแผนแคมเปญได้อย่างน้อย X แคมเปญภายใน 3 เดือนแรกหลัง Go-live
*   **SM3:** สามารถวัดผลได้ว่าลูกค้าที่ถูกระบุจาก RPT02 มีอัตราการตอบสนองต่อโปรโมชั่นสูงกว่ากลุ่มควบคุม (Control Group) หรือค่าเฉลี่ยในอดีต
*   **SM4:** ได้รับ Feedback เชิงบวกจาก Stakeholders เกี่ยวกับประโยชน์และความถูกต้องของข้อมูลที่ได้รับ
